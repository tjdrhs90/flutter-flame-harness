# flutter-flame-harness Phase B (Deploy) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the Phase B deploy skills (`admob`, `build`, `screenshot`, `submit`, `retro`) to the `flutter-flame-harness` plugin, taking a Phase-A-built game through AdMob integration, signed build + store upload, screenshots, metadata, and a retrospective — with the final store submission left manual (mirroring the user's proven fastlane pattern).

**Architecture:** Five new `SKILL.md` prompt documents reusing the Phase A machinery (context resets, file-based handoff in `docs/harness/`, `state.md` transitions). fastlane configs are NOT invented — they are generated by parameterizing the user's working setup at `<projects-dir>/flame_endless_runner/{ios,android}/fastlane`. Three surgical Phase A changes wire the new phases into the orchestrator + protocol + config.

**Tech Stack:** Markdown `SKILL.md`, Bash validators, fastlane (ruby 3.2) templates, Flutter `integration_test`, `google_mobile_ads`. Target: Flame 1.37 games.

## Global Constraints

- Conventional Commits; **NO AI-authorship trailers**. Commit identity: `git -c user.name='<your-name>' -c user.email='<support-email>'`.
- Plugin repo: `<projects-dir>/flutter-flame-harness/` (branch `main`). Push to `origin/main` at the end.
- Secrets (`credentials/`, per-game `secrets/`, `*.jks`, `*.p8`, `*.p12`, key `*.json`) never committed.
- iOS creds: `app_identifier com.<company>.<slug>`, `apple_id <support-email>`, `team_id <apple-team-id>`, ASC `key_id <asc-key-id>`, `issuer_id <asc-issuer-id>`, key file `AuthKey_<asc-key-id>.p8`.
- Android: `package_name com.<company>.<slug>`, `json_key_file fastlane/play-store-key.json`, signing alias `upload`, storeFile `upload-keystore.jks`, pw `<keystore-password>`.
- Submit boundary = **upload automated, final submit manual** (iOS → TestFlight + manual "Submit for Review"; Android → internal track + manual production promotion).
- iOS signing = ASC API key (`get_certificates` + `get_provisioning_profile`), NO fastlane match.
- Ads = rewarded-focused + iOS ATT + UMP consent; ad units created manually in AdMob console.
- Every phase skill: cite `docs/harness-protocol.md` for schemas (DRY); set `status: running` + `current_phase` + `next_role` on successful completion; append one row to `pipeline-log.md`.
- Reference fastlane (source of truth to mirror): `flame_endless_runner/ios/fastlane/Fastfile` (lanes: auth_check, certs, beta, metadata, categories, screenshots) and `flame_endless_runner/android/fastlane/Fastfile` (lanes: metadata, images, release_notes, internal).

**Interfaces:**
- Produces: parameterized fastlane templates (placeholders `__APP_ID__`, `__APP_NAME__`, `__IPA_NAME__`, `__PACKAGE__`, `__PROFILE_NAME__`) that the build skill (Task 4) fills per game; `scripts/validate-fastlane.sh` exits 0 only if both Fastfile templates are valid ruby after placeholders are substituted with dummy values.

- [ ] **Step 1: Write the failing fastlane validator**

Create `scripts/validate-fastlane.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
ROOT="$(cd "$(dirname "$0")/.." && pwd)"
fail=0
for tpl in "$ROOT/templates/fastlane/ios-Fastfile.template" "$ROOT/templates/fastlane/android-Fastfile.template"; do
  [ -f "$tpl" ] || { echo "FAIL: missing $tpl"; fail=1; continue; }
  # substitute placeholders with dummy values, then ruby -c
  sed -e 's/__APP_ID__/com.<company>.dummy/g' \
      -e 's/__APP_NAME__/Dummy/g' \
      -e 's/__IPA_NAME__/Dummy.ipa/g' \
      -e 's/__PACKAGE__/com.<company>.dummy/g' \
      -e 's/__PROFILE_NAME__/com.<company>.dummy AppStore/g' "$tpl" \
    | ruby -c 2>/dev/null | grep -q "Syntax OK" || { echo "FAIL: ruby syntax in $tpl"; fail=1; }
done
[ "$fail" -eq 0 ] && echo "validate-fastlane: OK" || exit 1
```
